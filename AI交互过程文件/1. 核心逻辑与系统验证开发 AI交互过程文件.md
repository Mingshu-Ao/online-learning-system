# AI 交互说明文档

## 1. 文档目的

本文档用于总结本次与 AI 协作完成“在线学习系统”需求分析、功能扩展、工程文档设计和 AI 辅助开发规划的交互过程。

通过本次交互，用户将一个初步的在线学习系统设想，逐步扩展为一套较完整的软件工程设计文档体系，并形成可供后续 AI 编码工具继续执行的开发提示词。

---

## 2. 交互背景

用户希望设计并开发一个在线学习系统。

该系统不仅包含传统在线教育平台的课程、视频、题库、测验、教师管理和后台管理功能，还希望加入更具创新性的模块，例如：

1. 线上自习室
2. WebSocket 实时状态同步
3. 番茄钟与打卡监督
4. AI 智能助教
5. 多模态错题解析
6. 知识图谱学习路径推荐
7. 学习数据分析与学情监控

用户同时提供了一份《AI 辅助软件工程指南》的文档模板，希望 AI 根据该模板，为项目补齐一系列 Markdown 工程文档。

---

## 3. 用户初始输入内容概述

用户首先提供了在线学习系统的基础设想，主要包括以下内容：

### 3.1 核心技术栈建议

用户提出系统采用如下技术路线：

- 后端：Spring Boot
- 实时通信：WebSocket
- 缓存与实时状态：Redis
- 数据库：MySQL
- 前端：Vue.js、React 或 Uni-app
- AI 层：PyTorch
- AI 服务部署：Docker 容器化独立微服务

### 3.2 系统核心模块

用户提出系统应包含三类端：

1. 学员端
2. 教师端
3. 管理端

其中学员端关注课程学习、视频播放、错题本和证书；教师端关注课程资源、题库和学情监控；管理端关注 RBAC 权限、课程审核、公告发布和日志审计。

### 3.3 核心亮点：线上自习室

用户特别强调线上自习室模块，需要实现：

- 虚拟座位
- 用户状态机
- 实时状态同步
- WebSocket 广播
- Redis 缓存在线状态
- 番茄钟
- 打卡契约
- 专注时长记录
- 打卡海报生成

### 3.4 创新点

用户提出两个重要创新方向：

1. **多模态 AI 智能助教与纠错**
   - 支持错题截图上传
   - 支持 OCR
   - 支持手写草稿识别
   - 支持 AI 生成解题思路和知识点扩展

2. **基于知识图谱的动态学习路径**
   - 将课程知识点构建为图谱
   - 根据错题率发现薄弱节点
   - 回溯前置知识点
   - 推荐基础视频和复习路径

### 3.5 数据库核心规划

用户提出初步数据库表包括：

- User
- Role
- Permission
- Course
- Chapter
- Resource
- Study_Record
- Study_Room
- Room_Record
- Question
- Paper
- User_Answer

### 3.6 非功能性需求

用户强调系统需要满足：

- 性能要求
- 安全要求
- 健壮性要求

例如高并发、低延迟、JWT 鉴权、密码加密、断网恢复等。

## 10. 本次交互体现的 AI 辅助开发方法

本次交互体现了以下 AI 辅助软件工程方法：

### 10.1 先文档后代码

在开始编码前，先明确：

- 需求
- 架构
- 接口
- 约定
- 测试
- 部署

这样可以降低 AI 编码时的随机性。

### 10.2 渐进式任务拆解

没有让 AI 一次生成整个系统，而是将开发拆成多个模块：

```text
用户权限 -> 课程资源 -> 学习进度 -> 测验考试 -> 自习室 -> AI 助教 -> 知识图谱 -> 管理端 -> 前端 -> 测试部署
```

### 10.3 明确控制边界

通过 `CONVENTIONS.md`、`CODINGSTYLE.md` 和各模块文档限制 AI：

- 哪些事情能做。
- 哪些事情不能做。
- 接口如何返回。
- Redis Key 如何命名。
- Controller 不能写业务逻辑。
- AI 服务不能直接改核心业务数据。

### 10.4 强调验证标准

通过以下文档保障可验证性：

- `UNITTEST.md`
- `集成测试标准.md`
- `系统测试标准.md`
- `边界情况.md`

使后续开发不只是“能运行”，还要“能测试、能验收”。

## 11. 对后续开发的建议

后续开发时，建议严格按照以下顺序推进：

```text
1. 创建项目骨架
2. 实现统一响应、异常处理、错误码
3. 实现用户登录与 RBAC
4. 实现课程与章节
5. 实现资源上传与访问鉴权
6. 实现视频学习进度
7. 实现测验考试与错题本
8. 实现线上自习室
9. 实现 AI 助教调用代理
10. 实现知识图谱推荐
11. 实现管理端
12. 实现前端页面
13. 补齐测试
14. Docker Compose 联调
```

每次给 Codex 的上下文应尽量控制在当前模块相关文档中，不要一次性提供所有文档。

例如开发自习室模块时，只需要提供：

```text
OUTLINE.md
REQUIREMENTS.md 中自习室部分
ARCHITECTURE.md 中 Redis 和 WebSocket 部分
CONVENTIONS.md
modules/module-study-room.md
Interface.md 中自习室接口和 WebSocket 协议
UNITTEST.md 中自习室测试要求
边界情况.md 中自习室边界情况
```
